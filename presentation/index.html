Lark Reading Calendar

把碎片化待读内容，变成可排期的阅读工作流

从微信剪贴板到 CloudBase 队列，再到飞书日历和多维表：让“稍后阅读”真正进入时间表。

观看项目介绍视频
入口
微信内容、网页链接、视频笔记
收集
iPhone 快捷指令 / OpenClaw / curl
调度
CloudBase 云函数与队列
落地
飞书日历、飞书多维表、反馈页

核心目标

不是再做一个收藏夹，而是把待读内容从“存起来”推进到“安排、阅读、反馈、复盘”。

Pain Point

收藏内容，稍后阅读=从此积灰，再也不读？

内容不是缺少入口，而是缺少一个能把它们推进下一步的系统。

入口分散

  • 微信、网页、视频、笔记各有各的收藏入口
  • 临时收藏很快变成长期堆积

没有时间约束

  • “稍后读”没有变成具体的日程安排
  • 没有提醒，也没有成批处理节奏

状态不可追踪

  • 读没读、读完没有、反馈是什么都散落在各处
  • 复盘和沉淀很难持续发生
Product

这个项目做了什么：一个跨设备的阅读收件箱

快捷指令复制微信内容，一键提交
CloudBase鉴权、去重、入队、凑批
飞书群提示收集进度和排期结果
飞书日历创建阅读 block
多维表记录状态、批次和反馈

它把“复制一下”这个低成本动作，接到了一个云端工作流上。

Technical Route

核心技术路径：CloudBase + 飞书 OpenAPI

为什么用 CloudBase

  • 作为云端状态中心，不依赖某台电脑在线
  • 手机、机器人、浏览器都能调用同一个 HTTP 入口
  • 数据库保存 items、batches、auth 三类状态

为什么用飞书 OpenAPI

  • 云函数直接创建日历事件、写入多维表
  • OAuth token 存在部署者自己的 CloudBase 里
  • 更适合跨设备、后台运行和长期自动化
Inspiration

和 Zara 的 reading-block-lark 是什么关系？

Zara 项目提供了 Reading Block 的清晰灵感：把零散待读链接成批写入飞书日历。

维度 Zara reading-block-lark Lark Reading Calendar
入口 Chrome 插件 快捷指令、机器人、curl
存储 浏览器本地 CloudBase 数据库
调用方式 本机 helper + lark-cli 云函数 + 飞书 OpenAPI
运行环境 依赖电脑和浏览器 云端运行，跨设备触发
状态管理 本地列表 队列、批次、多维表、反馈
Innovation

不是重写插件，而是把它扩展成工作流系统

入口扩展

从浏览器插件扩展到 iPhone 快捷指令、微信机器人和其他自动化入口。

状态扩展

从本地临时列表扩展到云端队列、批次状态、OAuth 授权和反馈记录。

模板扩展

快捷指令、多维表、CloudBase 配置和教程都做成可复制模板。

Open Source Boundary

开源不是把自己的项目直接丢出去

可以公开的

  • 云函数代码和依赖
  • CloudBase 配置模板
  • 快捷指令模板链接
  • 飞书多维表模板和教程文档

必须隔离的

  • 真实 webhook、App Secret、envId
  • 个人快捷指令配置
  • 用户部署后的 OAuth token
  • 复制后的多维表 app token / table id
What I Learned

真正重要的是数据流、权限流、运行环境

概念 要追问的问题 这个项目里的答案
数据流 内容从哪里来，存到哪里，最后写到哪里？ 快捷指令 → CloudBase 队列 → 飞书日历 / 多维表
权限流 谁有权限调用飞书？token 存在哪里？ 部署者自己 OAuth 授权，token 存入自己的 CloudBase
运行环境 它依赖本机在线，还是能在云端长期运行？ CloudBase 云函数负责后台入口和调度
Reflection

Vibe coding 让开始更快，理解原理让项目走得更远

AI 可以帮我更快搭出项目，但我仍然需要理解：数据在哪里、权限怎么走、系统边界在哪里、别人如何复用。真正的学习，是在实践里把这些问题一层层拆开。
